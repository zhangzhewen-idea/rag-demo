# RAG 评估模块操作手册

## 1. 手册用途

RAG 评估模块用固定的问题、黄金答案和黄金证据，回放当前知识库的检索与问答链路，帮助运营人员判断：

- 系统能否找到正确证据；
- 最终送给模型的证据是否完整、准确；
- 模型回答是否忠于证据、直接回应问题；
- 对知识库中没有依据的问题，系统能否正确拒答；
- 当系统配置、模型或知识内容变化后，质量是否明显下降。

评估结果是质量判断和回归对比的依据，不代表对所有真实用户问题的绝对保证。

## 2. 使用前提

开始评估前，请确认：

1. 当前账号具有管理员权限；非管理员无法访问评估页面。
2. 目标知识库已创建，相关文档已完成入库，且在正常问答中可用。
3. 已准备可核对的业务问题、标准答案和证据原文。
4. 评估会调用实际检索、重排、回答模型和 Judge 模型，因此需要相关服务正常，也会产生一定的处理时间和 Token 消耗。

登录运营后台后，在左侧“管理员”菜单中进入“RAG 评估”。

## 3. 评估集管理

### 3.1 什么是评估集

评估集是一组固定的评估样本，并与某一个知识库绑定。每条样本至少包含问题、黄金答案和答案类型；非拒答样本还必须包含黄金证据。

建议按稳定业务范围组织评估集，例如“员工制度核心问题”、“产品手册回归集”，不要将无关业务混在同一个评估集中。

### 3.2 创建评估集

1. 在页面右上角选择目标知识库。
2. 点击“新建评估集”。
3. 填写评估集名称和版本。同一知识库下，评估集名称与版本的组合不能重复。
4. 逐条填写评估样本；单个评估集最少 1 条、最多 100 条样本。
5. 确认所有字段后，点击“创建评估集”。

> 当前版本不支持编辑或删除已创建的评估集。如需修正问题、答案或证据，应创建新版本，例如从 `v1` 升级为 `v2`。

### 3.3 样本字段说明

| 字段 | 含义 | 填写建议 |
| --- | --- | --- |
| 问题 | 实际发送给 RAG 系统的用户问题 | 使用真实用户可能提出的自然语言，不要在问题中暗示文档位置 |
| 黄金答案 | 运营或业务专家认可的标准答案 | 简洁、明确、可核对，覆盖必须回答的要点 |
| 答案类型 | 当前问题的业务形态 | 从五种类型中选择，见下表 |
| 关键样本（失败时阻断门禁） | 该样本未通过时是否必须阻断整次评估 | 将合规、安全、核心业务规则等不允许出错的问题开启“失败时阻断门禁” |
| 黄金证据 | 知识库中应被检索到的来源文件和关键原文 | 非拒答样本至少填写 1 条；多文档或多要点问题可添加多条 |

### 3.4 答案类型

| 页面选项 | 系统标识 | 适用场景 | 示例 |
| --- | --- | --- | --- |
| 事实问答 | `FACTUAL` | 询问明确事实、数值、日期或规则 | “正式员工每年有几天年假？” |
| 步骤说明 | `PROCEDURE` | 询问办理流程、操作步骤或所需材料 | “如何申请差旅报销？” |
| 比较分析 | `COMPARISON` | 对比两个或多个制度、产品或方案 | “A 套餐和 B 套餐的适用对象有什么不同？” |
| 拒答 | `REFUSAL` | 知识库没有可支撑答案的问题 | “明天股价会涨多少？” |
| 摘要 | `SUMMARY` | 要求归纳文档、制度或多个要点 | “请概括新员工入职流程。” |

### 3.5 黄金证据的填写与匹配

每条黄金证据包含两项：

- **来源文件名**：必须与知识库中的文档来源名匹配。
- **证据中必须包含的文本**：使用能唯一标识答案依据的稳定原文片段。

系统的当前匹配规则为：

1. 来源文件名一致；
2. 检索到的切片文本包含填写的证据片段；
3. 匹配时忽略英文大小写，并将连续空白视为一个空格；
4. 标点、数字和其他文字差异不会被自动忽略。

填写时应直接从入库文档复制一段连续原文。避免填写整页长文，也不要将原文改写成同义句，否则可能出现人工看起来正确、系统却无法匹配的情况。

#### 普通问答样本示例

| 字段 | 示例内容 |
| --- | --- |
| 问题 | 正式员工每年有几天年假？ |
| 黄金答案 | 正式员工每年可享受 10 天年假。 |
| 答案类型 | 事实问答 |
| 失败时阻断门禁 | 开启 |
| 来源文件名 | `员工休假管理制度.md` |
| 证据中必须包含的文本 | “正式员工每年享有 10 天年假” |

#### 拒答样本示例

| 字段 | 示例内容 |
| --- | --- |
| 问题 | 明天公司股价会涨多少？ |
| 黄金答案 | 知识库中没有可靠依据，应明确拒绝预测。 |
| 答案类型 | 拒答 |
| 失败时阻断门禁 | 按业务风险决定 |
| 黄金证据 | 不填写 |

### 3.6 失败时阻断门禁

“失败时阻断门禁”是每条评估样本上的独立开关，用于标记该样本是否属于不允许出错的关键样本。

| 开关状态 | 样本未通过时的结果 |
| --- | --- |
| 开启 | 无论八项汇总指标最终是否达标，整次运行都会被判定为“质量门禁未通过” |
| 关闭 | 该样本不会单独强制阻断门禁，但它的分数仍会计入汇总指标，可能因汇总指标不达标导致整次运行未通过 |

建议在以下样本中开启：

- 涉及法律、合规、数据安全或人身安全的问题；
- 必须精确回答的核心制度、费用、时间、权限等规则；
- 不得误答的高风险拒答场景；
- 已发生过线上事故，必须防止再次回归的问题。

开启后，系统仍会继续执行评估集中的其他样本，并在全部样本处理完成后给出总体结论，不会在首个关键样本失败时立即停止任务。

> 样本执行异常会阻止整次运行通过，不受此开关影响。开关值会随评估集一同保存；如需修改，应创建新的评估集版本。人工复核不会改变该开关已产生的门禁结论。

## 4. 运行评估

### 4.1 启动评估

1. 选择知识库。
2. 在左侧“评估集”列表中选择目标名称和版本。
3. 点击“开始评估”。
4. 系统异步处理所有样本，页面每 2 秒自动刷新一次运行状态。

同一评估集同一时间只能有一个处于“排队中”或“评估中”的任务。当前任务结束前，“开始评估”按钮不可用。

运行开始时，系统会保存影响结果的配置快照，包括问答模型、Judge 模型、Embedding 模型、索引、检索、重排、切片和质量门槛等配置，便于后续定位变化。

### 4.2 运行状态

| 状态 | 页面显示 | 含义 | 运营动作 |
| --- | --- | --- | --- |
| `QUEUED` | 排队中 | 任务已创建，正在等待执行 | 等待页面自动刷新；长时间不变时联系研发 |
| `RUNNING` | 评估中 | 系统正在逐条处理样本 | 关注进度，不要重复启动 |
| `PASSED` | 已通过 | 运行满足当前质量门槛和基线回归规则 | 记录结论，必要时抽查关键样本 |
| `FAILED` | 未通过 | 运行完成，但存在指标、关键样本、基线回归或部分样本执行失败 | 展开样本结果，按第 8 章排查 |
| `ERROR` | 异常 | 运行未能形成有效评估结果，或发生运行级异常 | 记录运行编号和错误信息，确认依赖服务后重试或联系研发 |

### 4.3 基线规则

- 系统使用同一评估集最近一次 `PASSED` 运行作为对比基线。
- 评估集第一次运行没有基线，只判断绝对门槛。
- `FAILED` 和 `ERROR` 运行不会成为新基线。
- 新建评估集版本是独立的评估集，首次运行不与旧版本自动对比。

如果问题集、黄金答案、黄金证据或知识范围发生明显变化，建议创建新版本，不要将不同口径的结果直接与旧基线比较。

## 5. 评估指标说明

所有质量指标范围均为 0～1，页面以百分比显示，数值越高越好。页面汇总分是适用样本对应指标的平均值；不适用的指标显示为 `-`，且不参与该指标的门槛判断。

| 指标 | 计算对象与含义 | 当前默认门槛 | 低分通常表示 |
| --- | --- | ---: | --- |
| 候选命中率 | 对每条非拒答样本，候选检索结果中至少出现一条黄金证据则记为 100%，否则为 0%；汇总值是样本平均值 | 90% | 混合检索未找到关键内容，或黄金证据标注不正确 |
| 候选 MRR | 黄金证据在候选列表中首次出现位置的倒数：第 1 名为 100%，第 2 名为 50%，第 3 名约为 33.3%，未出现为 0% | 70% | 虽然找到了正确证据，但排名靠后，容易被后续截断 |
| 上下文召回率 | 最终证据覆盖的黄金证据数 ÷ 应覆盖的黄金证据总数 | 80% | 重排或 TopK 截断后遗漏了必要证据，对多证据问题尤其敏感 |
| 上下文精确率 | 最终证据中与黄金证据匹配的切片数 ÷ 最终证据总数 | 60% | 最终证据夹杂较多无关内容，会干扰模型回答 |
| 忠实度 | Judge 判断模型回答中的事实是否都能被最终证据支撑 | 80% | 回答包含证据之外的推断、编造或过度延伸 |
| 答案相关性 | Judge 判断回答是否直接、完整地回应问题，并符合所选答案类型 | 80% | 答非所问、要点不全、过度冗长，或拒答样本没有正确拒答 |
| 证据支持准确率 | Judge 判断系统关联的最终证据是否确实支持回答中的结论 | 80% | 展示的证据与回答结论对应不上，引用看似相关但不能完成证明 |
| 拒答准确率 | 仅对无黄金证据的拒答样本计算；最终证据为空时记为 100%，只要返回了证据就记为 0% | 90% | 系统对无依据问题仍检索出证据，存在误答风险 |

> 忠实度、答案相关性和证据支持准确率由 Judge 模型评分。Judge 仅依据问题、黄金答案、模型回答和最终证据评分，但模型评判仍可能存在小幅波动。不应仅凭单次小幅变化得出质量结论，应结合样本明细和人工复核判断。

## 6. 质量门禁规则

### 6.1 当前默认门槛

当前默认值为：

- 候选命中率 ≥ 90%；
- 候选 MRR ≥ 70%；
- 上下文召回率 ≥ 80%；
- 上下文精确率 ≥ 60%；
- 忠实度 ≥ 80%；
- 答案相关性 ≥ 80%；
- 证据支持准确率 ≥ 80%；
- 拒答准确率 ≥ 90%；
- 任一适用的汇总指标相对最近通过基线的下降不得超过 3 个百分点。

以上是项目当前的**默认值**。部署环境可通过环境变量覆盖，实际门禁判定以评估运行时加载的配置为准。若页面结论与上述默认值不一致，应请研发或运维核对当次运行的配置快照。

### 6.2 整体通过条件

一次运行标记为 `PASSED` 需同时满足：

1. 至少有 1 条样本执行完成；
2. 没有样本执行异常；
3. 没有关键样本未通过；
4. 所有适用的汇总指标达到绝对门槛；
5. 有基线时，任一适用指标相对基线的下降不超过 3 个百分点。

每条样本也会按绝对门槛显示“通过”或“未通过”。非关键样本单独未通过时，不一定直接导致整次运行失败；整体结论仍由汇总指标决定。关键样本只要有一条未通过，整次运行必定未通过。

## 7. 查看评估报告与人工复核

### 7.1 运行摘要

在“评估报告”中先查看：

- **质量门禁结论**：通过或未通过；
- **八项汇总指标**：用于定位主要问题环节；
- **样本进度和失败数**：区分质量不达标与样本执行异常；
- **P95 延迟**：95% 的已完成样本耗时不超过该值，用于观察慢请求；
- **Token**：当次回答生成累计的输入与输出 Token，用作资源消耗参考，不宜直接等同于最终账单；
- **基线编号**：当次运行用于回归对比的历史通过运行。

### 7.2 样本明细

点击展开某条样本，依次核对：

1. 问题与黄金答案是否正确；
2. 模型回答是否覆盖标准答案要点；
3. Hit、Recall、Precision、忠实度和证据准确率的样本分数；
4. “查看最终证据”中的来源、切片编号、相似度和证据原文；
5. Judge 给出的“裁判说明”；
6. 该样本的处理耗时和执行错误。

不要只看汇总分。检索类指标低时，优先核对最终证据；生成类指标低时，应同时对照模型回答、黄金答案和最终证据。

### 7.3 人工复核

1. 展开需要复核的样本。
2. 点击“人工复核”。
3. 选择“准确”或“不准确”，并填写复核备注。
4. 点击“保存复核”。

复核备注建议说明具体原因，例如“黄金答案已过期”、“证据完整但 Judge 误判”或“模型遗漏第二个步骤”，避免只填写“有问题”。

> 人工复核只保存运营判断和备注，不会修改样本的自动分数、通过状态或整次运行的质量门禁结论。已复核样本可以再次提交，新结论会覆盖原复核信息。

## 8. 异常和低分排查

| 现象 | 优先核对 | 建议处理 |
| --- | --- | --- |
| 候选命中率低 | 来源文件名、黄金证据原文、知识库文档是否已入库 | 先修正评估集标注；标注无误但反复检索不到时，将运行编号、问题和预期证据提供给研发 |
| 候选 MRR 低 | 候选命中率是否正常、最终证据是否仍包含正确内容 | 页面当前不展示完整候选排名；记录运行编号和问题，请研发核对候选顺序、检索权重和查询改写 |
| 上下文召回率低 | 候选阶段已命中、最终证据却缺失的内容 | 核对是否过度标注了黄金证据；标注无误时请研发检查重排和 TopK 截断 |
| 上下文精确率低 | 最终证据中的无关切片 | 记录无关来源和切片，请研发检查相似度阈值、重排和 TopK |
| 忠实度低 | 回答中哪些结论无法从证据找到 | 人工标注具体幻觉内容；反复出现时请研发检查上下文组装和回答提示词 |
| 答案相关性低 | 问题、黄金答案和答案类型是否匹配，回答是否漏项或答非所问 | 先修正错误的评估集定义；定义无误时标注漏答内容并联系研发 |
| 证据支持准确率低 | 最终证据是否真正支持回答结论 | 区分“证据选错”和“模型误用证据”，在复核备注中写明 |
| 拒答准确率低 | 拒答样本是否真的不应有证据，系统实际返回了哪些证据 | 如知识库已新增相关内容，应创建新评估集版本；否则请研发检查误召回 |
| 指标小幅波动 | Judge 说明、具体样本和多次运行是否一致 | 可在外部服务稳定时重跑一次；不要用反复重跑代替问题定位 |
| 相对基线下降超过 3 个百分点 | 基线编号、知识库内容、模型和检索配置是否变更 | 将变更时间与失败样本一并提供给研发；如评估口径本身已改变，创建新版本 |
| 任务长时间处于排队中或评估中 | 页面网络、后端服务、评估运行线程和外部 AI 服务 | 先刷新页面确认真实状态；仍无变化时记录运行编号、开始时间并联系研发 |
| 运行显示“异常” | 页面错误信息，以及检索、重排、问答模型、Judge、数据库等服务是否可用 | 记录运行编号、发生时间和完整错误信息；确认服务恢复后重试，反复出现时联系研发 |

### 联系研发时应提供的信息

- 知识库名称；
- 评估集名称和版本；
- 运行编号和基线编号（如有）；
- 发生时间和页面错误信息；
- 未通过的样本问题；
- 预期证据与实际最终证据；
- Judge 裁判说明和人工判断。

## 9. 运营执行建议

### 9.1 评估样本覆盖建议

评估集应结合实际业务逐步覆盖：

- 高频用户问题；
- 核心业务规则和必须正确的关键问题；
- 口语化、同义改写和简称问题；
- 中英文混合、专有名词和精确数值问题；
- 需要综合多条或多份文档的问题；
- 与正确答案相似但不应命中的干扰场景；
- 知识库无法回答、必须拒答的问题。

### 9.2 建议执行时机

- 知识库核心文档更新后；
- 检索、重排、切片、模型或提示词配置变更后；
- 重要版本发布前；
- 线上出现典型错答后，先将该问题补充到新评估集版本，再运行回归。

### 9.3 每次评估检查清单

- [ ] 选择了正确的知识库和评估集版本。
- [ ] 确认知识库文档已完成入库。
- [ ] 确认高风险样本已开启“失败时阻断门禁”。
- [ ] 查看了运行终态和质量门禁结论。
- [ ] 对比了八项汇总指标与基线。
- [ ] 展开了未通过和关键样本。
- [ ] 核对了模型回答、最终证据和 Judge 裁判说明。
- [ ] 对有争议的样本完成了人工复核并写明原因。
- [ ] 异常或持续低分时，已记录运行编号、时间、样本和证据信息。
