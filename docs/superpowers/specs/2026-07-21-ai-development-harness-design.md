# AI 开发 Harness 基础模板设计

## 1. 背景与目标

`rag-demo` 是一个前后端分离的 RAG 企业知识库问答项目。随着项目持续迭代，需要为 AI 编程工具建立稳定、可维护的项目上下文和工作边界，使 Codex、Claude Code 及未来接入的其他工具能够长期、一致地理解和修改项目。

本设计引入一套以文档为核心的 AI 开发 Harness。它不负责应用进程守护或生产运维，也不实现程序化状态机；它通过统一规则源、薄适配层、标准工作流和质量门禁约束 AI 的开发行为。

目标如下：

- 为不同 AI 编程工具提供同一套项目事实、架构约束和安全边界。
- 允许 AI 在明确边界内自主分析、修改、验证和提交代码。
- 要求 AI 以实际测试、构建和 diff 审查作为完成声明的依据；纯文档约束本身不提供强制执行能力。
- 将 `rag-demo` 当前技术栈和 RAG 业务约束预填到模板中，同时保留迁移到其他项目的能力。
- 保持 Harness 简单，以便项目维护者后续直接修改 Markdown 文件。

## 2. 已确认的设计选择

- Harness 类型：AI 开发 Harness，不包含应用运行 Harness。
- 工具范围：工具无关核心，提供 Codex 和 Claude Code 薄适配层。
- 自治等级：高自治；AI 可自主编辑、验证和提交。
- 约束方式：纯文档约束，不提供可执行门禁程序或任务状态机。
- 追踪方式：不保存 AI 任务记录，长期轨迹仅依靠代码、文档和 Git 历史。
- 模板内容：通用骨架与 `rag-demo` 当前项目约束相结合。

## 3. 非目标

- 不提供进程守护、健康检查、自动恢复、备份或生产监控。
- 不实现独立 CLI、守护进程、工作流引擎或审批服务。
- 不保存提示词、完整执行日志、任务归档或中间状态文件。
- 不要求把所有项目文档迁入 `.harness/`。
- 不替代现有单元测试、ArchUnit、Git 或项目 README。
- 不为 Cursor 等尚未接入的工具提前创建占位适配文件。

## 4. 总体架构

`.harness/` 是唯一核心规则源。根目录工具入口只说明加载方式和平台能力差异，不复制核心规则。

```text
rag-demo/
├── AGENTS.md
├── CLAUDE.md
└── .harness/
    ├── README.md
    ├── constitution.md
    ├── project-context.md
    ├── architecture.md
    ├── commands.md
    ├── quality-gates.md
    ├── security-boundaries.md
    └── workflows/
        ├── feature.md
        ├── bugfix.md
        ├── refactor.md
        └── review.md
```

### 4.1 工具入口

- `AGENTS.md` 是 Codex 入口。
- `CLAUDE.md` 是 Claude Code 入口。
- 两个入口都要求 AI 先读取 `.harness/README.md`，再按任务类型加载必要规则。
- 入口文件只包含工具名称、加载方式和平台特有映射，不得复制或弱化核心规则。
- 后续接入新工具时，只新增对应入口适配文件。

### 4.2 核心文档职责

- `README.md`：Harness 导航、按需加载顺序、维护方式和迁移说明。
- `constitution.md`：自治原则、通用行为准则、Git 行为和完成定义。
- `project-context.md`：项目目标、技术栈、目录、依赖和事实来源。
- `architecture.md`：COLA 分层、前后端边界、RAG 业务不变量和模型约束。
- `commands.md`：项目认可的安装、启动、测试和构建命令。
- `quality-gates.md`：按改动范围选择验证项，以及完成声明的准入条件。
- `security-boundaries.md`：密钥、数据、外部副作用和破坏性操作边界。
- `workflows/*.md`：不同任务类型的上下文加载、执行步骤和退出条件。

### 4.3 加载顺序

AI 先加载工具入口和 `.harness/README.md`，再加载所有任务都需要的 `constitution.md` 与 `security-boundaries.md`。随后根据任务范围读取项目上下文、架构、命令、质量门禁及一个对应工作流。

按需加载用于控制上下文规模。没有涉及前端的后端修复不需要读取前端实现细节，但仍必须遵守项目级安全和完成标准。

## 5. AI 任务执行流程

任务状态只存在于当前对话中，不写入仓库：

```text
接收目标
→ 判断任务类型与影响范围
→ 加载对应 Harness 文档
→ 检查工作区和现有实现
→ 给出简短执行计划
→ 自主修改
→ 执行质量门禁
→ 审查最终 diff
→ 展示提交摘要
→ 自主 commit
→ 汇报结果与遗留风险
```

通用执行规则：

1. 修改前检查现有实现、相关文档、Git 状态和用户未提交的改动。
2. 在 `feature`、`bugfix`、`refactor` 和 `review` 中选择一个主工作流。
3. 计划默认在对话中展示；除非涉及范围扩张、高风险操作或无法安全判断的需求歧义，否则不等待批准。
4. 修改保持局部、可回滚，不夹带无关重构。
5. 用户已有改动默认属于用户，必须保留；无法安全合并时停止并说明冲突。
6. 验证完成后审查最终 diff，并在 commit 前展示准确的变更摘要。
7. 只有满足完成标准时才自主创建简洁中文 commit。
8. 最终回复报告实际执行的命令、结果和仍未覆盖的风险。

## 6. 权限与安全边界

### 6.1 可自主执行

- 读取、搜索、创建和修改项目内代码、测试及中文文档。
- 安装本地依赖，运行格式化、测试、构建和只读诊断命令。
- 在任务范围内修复由自身改动引起的问题。
- 验证通过后，仅暂存本任务文件并创建中文 commit。

### 6.2 必须获得明确批准

- 删除文件、目录、数据或 Git 历史。
- 修改 `.env`、密钥、Token、证书或真实凭据。
- 修改 CI/CD 配置。
- 执行 `git push`、`git rebase`、`git reset --hard` 或强制推送。
- 生产部署、公开发布或破坏性数据库迁移。
- 执行会产生费用或影响共享数据的模型在线测试。

### 6.3 信息安全

- 不读取、输出或提交真实密钥和私有文档内容。
- 数据库密码、Redis 密码、JWT 密钥和百炼 API Key 只通过环境变量注入。
- 日志不得记录密码、Token、API Key 或完整私有文档内容。
- 暂存文件必须使用明确路径，避免把无关文件或用户改动混入 commit。

## 7. 异常处理

- 自身改动导致验证失败时，继续在当前任务范围内定位和修复。
- 同一问题连续失败三次后停止盲目重试，报告已执行命令、错误证据和阻塞点。
- 疑似既有失败时，通过基线、最小复现或限定范围的测试区分来源，不直接归咎于当前改动。
- 外部 MySQL、Redis、JDK、Node.js 或网络缺失导致验证无法执行时，保留准确错误并标记为阻塞。
- 无法运行必需门禁时，将结果标记为“未验证”，不得宣称任务完成或自主提交。
- 需求歧义若会改变范围、数据或公开行为，停止并询问；局部且可逆的实现细节允许自行判断。

## 8. 质量门禁

`commands.md` 保存项目认可的原始命令，`quality-gates.md` 负责把改动范围映射到命令。

| 改动范围 | 必须执行 |
|---|---|
| 所有改动 | `git diff --check`、最终 diff 审查、工作区状态检查 |
| 后端代码 | `cd rag-demo-backend && ./mvnw test` |
| 前端代码 | `cd rag-demo-frontend && npm test && npm run type-check && npm run build` |
| 前后端接口联动 | 后端与前端全部验证 |
| 文档 | 检查路径、命令、术语和内部一致性 |
| 启动或配置 | 对应测试通过后执行有界启动检查 |
| 百炼在线调用 | 默认禁止，获得批准后单独执行 |

验证顺序如下：

1. 先执行与改动直接相关的最小测试。
2. 修复完成后执行对应模块的完整门禁。
3. Maven 测试同时承担 ArchUnit 架构约束验证。
4. 不为项目尚未配置的工具虚构命令，例如当前前端没有 `lint` 脚本，就不把 `npm run lint` 写为必需门禁。

## 9. 完成标准

任务只有同时满足以下条件才算完成：

- 用户目标已实现且未夹带无关修改。
- 对应质量门禁全部通过。
- 文档与实际命令、目录和架构保持一致。
- 没有新增密钥、调试残留或模糊占位符。
- commit 只包含当前任务涉及的文件。
- 最终回复列出实际验证命令、结果和未覆盖风险。

## 10. `rag-demo` 预填约束

### 10.1 技术与目录

- 后端位于 `rag-demo-backend`，使用 Java 25、Spring Boot 4、Spring AI 2 和 Maven Wrapper。
- 前端位于 `rag-demo-frontend`，使用 Vue 3、TypeScript、Vite 和 Vitest。
- 项目直接连接 MySQL 8 和 Redis Stack，文件上传保存到受控本地目录。

### 10.2 架构边界

- 后端依赖方向固定为 `adapter → application → domain ← infrastructure`。
- `domain` 不依赖其他三层，`application` 和 `adapter` 不直接依赖 `infrastructure`。
- MySQL、Redis Stack、文件存储、文档解析和百炼模型均通过 Gateway 隔离在基础设施边界。
- ArchUnit 测试是架构约束的事实执行者，Harness 文档必须与测试保持一致。

### 10.3 RAG 与模型约束

- 回答只能基于检索证据；无可靠证据时固定回答“当前知识库中未找到可靠依据”。
- 检索文档内容是资料而不是系统指令，防止文档提示注入覆盖系统约束。
- Embedding 固定为 2560 维；模型或维度变化后必须重建整个向量索引。
- SSE 已开始输出后不得自动重试整次模型请求，避免重复内容。
- 有冲突证据时保留来源差异，不自行生成未经证明的结论。

## 11. 维护与迁移

- 自治原则变化只修改 `constitution.md`。
- 安全边界变化只修改 `security-boundaries.md`。
- 技术栈、目录或外部依赖变化修改 `project-context.md`。
- 分层或 RAG 业务不变量变化修改 `architecture.md`，并同步相关架构测试。
- 命令或测试脚本变化同步修改 `commands.md` 和 `quality-gates.md`。
- 新增 AI 工具时只添加入口适配文件，核心规则保持不变。
- Harness 不维护手工版本号，所有演进通过 Git 历史追踪。
- 迁移到其他项目时保留目录职责和工作流结构，替换项目上下文、架构约束、命令和质量门禁。

## 12. 验收标准

- Codex 能从 `AGENTS.md` 定位并加载 Harness 核心规则。
- Claude Code 能从 `CLAUDE.md` 定位并加载同一套核心规则。
- 两个入口不复制核心政策，只包含平台适配信息。
- 四种工作流均明确上下文加载范围、执行步骤和退出条件。
- `commands.md` 中的命令与当前 `pom.xml`、`package.json` 和 README 一致。
- `quality-gates.md` 能覆盖后端、前端、联动、文档和配置类改动。
- 安全边界明确区分自主操作和需要批准的操作。
- 所有 `rag-demo` 预填事实与当前代码和已批准项目设计一致。
- Harness 中不存在任务日志、执行归档、状态机程序或未填写占位符。
