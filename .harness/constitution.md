# 开发宪章

## 自治原则

- 在用户目标和安全边界清晰时，自主完成分析、局部修改、验证、diff 审查和提交，不因普通实现细节等待批准。
- 计划保存在当前对话中，不向仓库写入任务状态、中间日志或提示词。
- 以当前代码、配置、测试和用户已批准设计为事实依据；文档与实现冲突时明确指出，不凭猜测统一口径。
- 保持改动局部、可回滚，不夹带无关格式化、清理或重构。
- 用户已有改动默认属于用户，必须保留；无法安全合并时停止并说明冲突。

## 通用执行准则

1. 修改前检查 Git 状态、现有实现、相关设计和影响范围。
2. 从 `feature`、`bugfix`、`refactor`、`review` 中选择一个主工作流。
3. 给出简短计划；只有范围扩张、高风险操作或会改变数据和公开行为的歧义才等待批准。
4. 先做与目标直接相关的最小改动和最小验证，再执行对应完整门禁。
5. 自身改动导致失败时继续定位和修复；同一问题连续失败三次后停止盲目重试，保留命令、错误和阻塞证据。
6. 疑似既有失败时通过基线、最小复现或限定范围测试区分来源，不直接归咎于当前改动。
7. 完成前审查最终 diff、检查工作区状态，并确认没有密钥、调试残留、模糊占位符或无关文件。

## Git 行为

- 不覆盖、不丢弃或顺手提交用户已有改动。
- 暂存时使用明确文件路径，不使用可能混入无关内容的宽泛操作。
- 提交前向用户展示准确的变更摘要。
- 只有目标实现且所有必需门禁通过时，才创建简洁中文 commit。
- 未验证、验证失败或存在未解决阻塞时不得提交，也不得宣称完成。
- `push`、`rebase`、改写历史等操作遵守 `security-boundaries.md`，提交授权不等于推送授权。

## 完成定义

任务仅在以下条件全部满足时完成：

- 用户目标已实现，且没有夹带无关修改。
- 改动范围对应的质量门禁全部通过。
- 文档与实际目录、命令、配置和架构一致，已识别差异被明确记录。
- 没有新增真实密钥、私有数据、调试残留或未填写占位符。
- commit 只包含当前任务文件。
- 最终回复列出实际执行的验证命令、结果和仍未覆盖的风险。

纯 Markdown 规则不能替代测试、ArchUnit、构建和人工 diff 审查；完成声明必须以实际检查结果为依据。
