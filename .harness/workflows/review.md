# 代码审查工作流

## 上下文加载

必读 `constitution.md`、`security-boundaries.md`、`project-context.md`、`architecture.md` 和
`quality-gates.md`，再读取待审 diff、相关调用路径、测试、配置与设计。存在 `.codegraph/` 时先检查变更符号的调用链和
动态实现；只有需要复现时加载 `commands.md` 中对应命令。

## 执行步骤

1. 明确审查基线和范围，检查 staged、unstaged、untracked 状态，不修改待审内容。
2. 沿调用路径检查正确性、回归、安全、数据一致性、并发、接口兼容和 COLA 依赖方向。RAG 变更分别审查查询处理、
   候选召回、最终证据、上下文、生成与评估口径，防止阶段概念混用。
3. 用相关测试或最小复现验证高风险判断；检查模型/重排失败降级、摘要并发、SSE 取消、文档状态 CAS 和删除补偿；区分
   确定缺陷、风险和风格偏好。
4. 按严重程度输出发现，每项给出文件、位置、触发条件和实际影响；结论先于摘要。
5. 若没有发现，明确说明，并列出未覆盖的测试或环境风险。
6. 只有用户明确要求修复时，切换到 `bugfix` 或 `refactor` 工作流后再修改和提交。

## 退出条件

- 已覆盖待审变更及其关键调用方、测试和架构边界。
- 涉及 migration、索引、真实评估运行或日志时，已单列数据、安全、费用和隐私风险。
- 每个发现都可定位、可解释并有证据，不把猜测当结论。
- 纯审查不修改、不暂存、不提交代码；若用户授权修复，则按切换后的工作流退出条件完成。
