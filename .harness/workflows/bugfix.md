# 缺陷修复工作流

## 上下文加载

必读 `constitution.md`、`security-boundaries.md`、`project-context.md`、`commands.md` 和
`quality-gates.md`。涉及分层、接口或 RAG 行为时同时读取 `architecture.md`；再读取故障路径、相关测试和日志结构。
存在 CodeGraph 索引时，先用它定位入口、Gateway 实现和动态调用路径。

## 执行步骤

1. 检查 Git 状态，使用实际命令、最小输入或现有测试复现问题并保存准确错误。
2. 判断失败来自当前代码、既有环境还是外部依赖；RAG 故障还要定位到改写、扩展、候选召回、RRF、重排、上下文、
   生成或 SSE 阶段，未经证据不猜测根因。
3. 添加能在修复前失败的最小回归测试；无法自动化时说明替代验证方式。
4. 修复根因，保持改动局部，不借机重构无关代码。重排或模型故障优先验证既有降级路径，不能用放宽全局阈值掩盖问题。
5. 运行回归测试和受影响模块完整门禁；同一问题连续失败三次后停止盲目重试。
6. 审查最终 diff、工作区和回归覆盖，展示提交摘要，只提交修复相关文件。

## 退出条件

- 已复现并有证据证明原故障不再出现，或明确记录环境阻塞。
- Flyway checksum、Redis 索引或文档删除故障只完成只读诊断时，不得擅自 repair、重建或删除数据。
- 回归测试和对应完整门禁通过，未破坏架构与业务不变量。
- 验证通过时已创建范围准确的中文 commit；未复现、未验证或仍失败时不提交、不宣称完成。
