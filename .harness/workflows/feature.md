# 新功能工作流

## 上下文加载

必读 `constitution.md`、`security-boundaries.md`、`project-context.md`、`architecture.md`、`commands.md`
和 `quality-gates.md`，再读取功能涉及的设计、现有实现、测试和接口调用方。仓库存在 `.codegraph/` 时，先通过
CodeGraph 确认入口、动态实现和调用路径。

## 执行步骤

1. 明确用户可观察目标、影响模块、接口和数据边界，检查 Git 状态及已有改动。
2. 找到最接近的现有用例和测试，给出简短计划；范围清晰时直接执行。
3. 先定义或补充能证明行为的最小测试，再按 COLA 职责放置实现；新增 RAG 能力要明确它属于候选召回、最终证据、
   上下文还是生成阶段。
4. 涉及外部能力时先定义或复用 `domain/gateway`，在 `infrastructure` 实现；adapter 只暴露 application
   用例。
5. 接口联动时同步后端 DTO、前端类型/API/调用方和契约测试；SSE 只扩展具名事件并保持取消/错误路径可收敛。
6. 新增外部模型、索引字段、评估指标或配置项时，同步配置绑定、默认值、运行快照、文档和降级测试；索引迁移、在线调用
   或数据变更先取得批准。
7. 先运行最小测试，再执行完整质量门禁，修复自身改动引起的问题。
8. 审查最终 diff 和工作区，展示提交摘要，只提交本功能文件。

## 退出条件

- 功能目标和异常路径有实现及相应验证。
- 架构、RAG、安全和接口不变量未被破坏。
- 检索类功能已分别证明候选召回和最终证据行为；在线评估未获批准时不运行且不伪装为自动化门禁。
- 对应门禁全部通过，diff 无无关改动，并已创建仅包含当前任务的中文 commit。
- 若需要扩大范围、执行受限操作或必需门禁无法运行，则不提交，报告证据并等待用户决策。
