# 质量门禁

## 范围映射

| 改动范围      | 必须执行                                                                      |
|-----------|---------------------------------------------------------------------------|
| 所有改动      | `git diff --check`、最终 diff 审查、`git status --short`                        |
| 后端代码或后端依赖 | `cd rag-demo-backend && ./mvnw test`                                      |
| 前端代码或前端依赖 | `cd rag-demo-frontend && npm test && npm run type-check && npm run build` |
| 前后端接口联动   | 后端和前端全部门禁，并检查 DTO、类型、调用方与 SSE/错误契约                                        |
| 文档        | 检查路径、命令、版本、术语、链接和内部一致性                                                    |
| 启动或运行配置   | 对应模块测试通过后执行有界启动与健康/可达性检查                                                  |
| Docker 发布资产 | `bash -n deploy.sh`、使用示例配置执行 `docker compose config`、构建前后端镜像；未经批准不启动容器        |
| 架构边界      | 后端 Maven 测试中的 ArchUnit 通过，并人工核对依赖方向                                       |
| 检索/上下文策略  | 对应 domain/application/infrastructure 测试；分别核对候选、最终证据和 Token 预算                   |
| RAG 评估代码   | 评估指标、阈值、服务/Worker、持久化与前端评估页面测试；普通门禁不得发起真实评估运行                       |
| Flyway migration | 检查是否已执行、校验 checksum 风险；禁止为格式化修改历史 migration                             |
| 百炼在线调用    | 默认禁止；获明确批准后单独执行，不以在线结果替代确定性测试                                        |

只修改 Harness Markdown 和工具入口时，按文档范围执行，不因文档提及前后端而运行全部应用测试；必须逐项核对其引用的构建脚本、目录、配置和架构测试。

## 验证顺序

1. 先运行与改动直接相关的最小测试或静态检查。
2. 修复完成后运行改动范围对应的完整门禁。
3. 运行 `git diff --check`，审查最终 diff 和工作区状态。
4. 区分当前改动引起的失败、既有失败和外部环境阻塞。
5. 所有必需门禁通过后，才展示提交摘要并提交当前任务文件。

## RAG 分阶段门禁

- 查询改写/扩展：验证普通问题、指代追问、空或异常模型响应的降级行为。
- 混合召回：验证 `knowledgeBaseId`/`documentId` 过滤、BM25 与向量排名、权重和 RRF 去重。
- 重排：验证成功排序及关闭、超时、限流、无效响应时的候选顺序回退。
- 上下文：验证摘要乐观锁、摘要失败回退、近期消息窗口、Token 上限和多来源证据保留。
- 评估：固定样本分别断言候选 `Hit Rate@K/MRR` 与最终 `Context Recall/Precision`；不得通过评价最终 LLM 文案
  反推召回质量。
- 生成：验证无证据固定拒答、引用只来自最终上下文、SSE 事件契约和取消/异常路径。

## 结果判定

- **通过**：必需命令成功，最终 diff 与用户目标一致，工作区没有被误改。
- **未验证**：必需命令因 JDK、Node.js、MySQL、Redis、网络或其他环境缺失无法运行。不得宣称完成或自主提交。
- **失败**：测试、构建、一致性或 diff 检查不通过。自身改动导致时继续修复；连续三次同类失败后停止重试并报告证据。

最终回复必须列出实际执行的命令及结果。未运行的门禁不得用“应当通过”替代，也不得因纯文档约束而省略可执行验证。
在线评估未获批准时应明确标为“按安全边界未运行”，不能标为环境阻塞。
